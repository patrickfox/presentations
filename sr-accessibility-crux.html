What is accessibility (a11y)?

For our clients:

  • Inclusivity
  • More sales, participation, improved brand perception
  • Avoiding lawsuits

For Razorfish:

  • Create efficient and innovative web experiences for all
  • Exceed the accessibility standards and best practices
  • Value-add, key differentiator
  • Being a good steward of the web
  • Looking out for our clients

For our end users:

  • Access to the conveniences of the web (e.g. banking, shopping, social)
  • Independence, self-reliance, participation

In a nutshell:

  • What can be seen or heard (color, imagery, icons, font weight/size, audio alerts) is read/interpreted by a screen reader (and still makes sense)
  • What can be heard is also indicated to the user via visible text
  • Any functions and interactions available via a mouse are also available via keyboard

Target Audience

Vision impairments

Partial to total blindness

  • Color blindness
  • Total blindness (only ~3% of blindness)

Hearing impairments

Partial to total deafness

  • Hard of hearing
  • Situational impairment (e.g. loud environment)

Mobility impairments

Loss of mobility, dexterity in arms or hands

  • Mobility control
  • Amputee or paralysis

Cognitive impairments

Ease of distraction or difficulty

  • Attention deficit disorder
  • PTSD
  • Brain injury

Areas of difficulty

From WebAIM's Annual Screen Reader Survey (2014)

  • Inaccessible Flash content
  • CAPTCHA’s
  • Ambiguous links
  • Missing/improper alt text
  • Unexpected screen changes
  • Complex/difficult forms
  • Poor keyboard accessibility
  • Missing or improper headings
  • Too many links
  • Complex data tables
  • Lack of ‘Skip’ links
  • Inaccessible/missing search

Screen Readers

Software that reads content, provides tools to navigate and understand UI

Top screen readers

  • JAWS
  • NVDA
  • VoiceOver (MacOS and iOS)
  • TalkBack (Android)

Basic Voiceover Commands

  • ⌘-F5 - toggle on/off
  • VO(ctrl + option) + left/right arrows - navigate through elements
  • VO + U - use arrows to view available elements
  • VO + A/B - start reading page from cursor or from top of page to cursor

Laws and Standards

Section 508 amendment to the Rehabilitation Act

Established a US government accessibility standard, requires that all US federal agencies ensure that their electronic and information technology are accessible per the Section 508 guidelines

WCAG2

W3C's international accessibility standard

  • levels of compliance: A, AA, AAA
  • WCAG2 Level AA covers the majority of Section 508

ARIA Specification

  • International, W3C spec
  • Addresses accessibility issues pertaining to dynamic and custom web UI
  • Defines a set of HTML attributes e.g. aria-label, role, aria-live

American's with Disabilities Act of 1990 (ADA)

A civil rights law that is intended to protect against discrimination based on disability

Specific to physical

National Federation of the Blind (NFB)

Blind and vision-impaired rights advocacy group, assists with high-profile lawsuits

Air Carrier Access Act (ACAA)

Mandates compliant, accessible web experiences for air carriers operating in the US by December 2015/2016

Accessibility for Ontarians with Disabilities Act, 2005 (AODA)

Canadian standard based on WCAG2 AA

Mandates timeline for government and public sector organization compliance

    Headings

    Heading Usage

    Wireframes prescribe heading and heading level usage

    One H1 Per Page

    One H1 is prescribed per view defined in the wires

    Modal headings should start at H1

    Consistent Use of Headings

    Heading styles are prescribed in design comps and convey heading level and hierarchy in a consistent manner across a site or application

    Sequential Headings

    Heading level is prescribed in the wires and is used to accurately indicate relationship and content hierarchy

    • Heading order should never increase by more than 1
    • Good: H2 > H3, H1 > H2, H4 > H2
    • Not so good: H2 > H4, H1 > H4

    Heading order is used to indicate hierarchy. An H3 is considered to be a child of its parent H2.

    Text Only Headings

    Design comps prescribe headings that do not contain clickable text

    Landmarks

    Purpose

    Landmarks designate key sections of the UI and improve understanding and navigation for screen reader users

    Landmark Rules

    All page content should live within a landmark - avoid orphaned content

    Use landmarks minimally - should designate high-level structure only

    Role Usage Description
    banner 1 Header section
    main 1 Main content section
    contentinfo 1 Footer content section
    navigation 0+ generic, container that houses navigation
    complementary 0+ generic, secondary content related to main view content
    search 0+ use role on enclosing search box container that contains search field; do not use directly on a form field

    Landmark Usage

    UX: Landmark usage is clearly defined in the wires

    Label Generic Landmarks

    Copy: Multiple generic landmarks (e.g. navigation, complentary) are labelled to differentiate their purpose, and label copy is captured in the copy deck.

    Tables

    Table Row/Column Headings

    UX/Copy: Wireframes prescribe table column and row headings that accurately represent their purpose of their column or row content.

    Complex Table Summary

    UX: Wireframes specify a table caption that provides a summary.

    General Copy

    Alternative Text

    Provide alternative text when visible text for labelling a button, link, form element or pertinent image is insufficient for understanding its purpose or context.

    Examples:

    • "Read More" links
    • Pertinent images
    • Visual indicators (progress bar, icons, red error text)

    Page Purpose

    Page Context Cues and Consistency

    Design: Current navigation item is indicated visually and in a consistent manner.

    Dynamic Content

    Dynamic Changes Announced

    UX: Wireframes document how to communicate dynamic UI updates to AT users.

    Dynamic Search / Filtering / Sorting

    UX/Copy: Dynamic search provides summary of results that informs AT users of change in content

    Example: https://www.thefreshmarket.com/search

    Timeout Warning and Extension

    UX: Wireframes document a mechanism (typically a modal dialog) that interrupts and alerts the user to an impending session timeout. In addition, the UI must allow the user to adjust the timeout in one of the following ways:

    • Turn Off - disable or opt out of the timeout option
    • Adjust - set a custom timeout limit
    • Extend - extend the time and that allow the user to extend the session for 15 minutes, up to 4 times, before forcing a timeout

    Infinite Scrolling

    UX: Wireframes prescribe an explicit means for users to load additional content (e.g. activating a “Load more results” button).

    Dynamic Content

    Form Controls are Labelled

    Design: Form controls have a visually-associated text label that is separate from placeholder or instructional text and is located before the form control (e.g. above or to the left).

    Placeholder Text Is Not a Label

    Design: Text that 1) appears by default inside of a text input and 2) disappears when a value is entered does not suffice as a label.

    Labels Contain Text Only

    Design: Labels defined in comps contain text only and no buttons or links.

    Required Form Controls

    Design: Comps clearly indicate the display of required fields.

    Formatting, Error and Help Messages

    Design: Pertinent instructional, error, formatting and help messages are present and are visually associated with their form controls.

    Communicate Generic Errors

    Design: Comps document the display of generic form errors, summarized and grouped prominently above the form.

    Reduce Distraction

    Provide Pause/Stop/Start Controls

    Design: Design comps document user-operable means to control any motion, animations and animated content (e.g. background videos, automated carousel content).

    No Blinking or Flashing Content

    Design: Design comps do not prescribe any blinking or flashing content.

    Keyboard Navigation

    Visible Keyboard Focus

    Design: Focus state for each interactive element type (e.g. buttons, links, form controls) is captured in the design comps.

    UI Consistency

    Same Relative Order

    Design: Design comps present a natural, logical and consistent flow for all components on a page, across the site.

    Consistent Responsive Experience

    Comps present content and controls as consistently as possible from breakpoint to breakpoint.

    Visual Design

    Text Color Contrast Ratio

    The comps provide a style guide that documents and prescribes valid, color contrast compliant text-to-background color combinations. This guide provides color hex values and contrast ratios, and serves as a usage guide for developers and a confirmation of accessibility compliance for QA.

    Examples:

    • Text superimposed over dynamic image or video
    • Requirement: text-to-background color contrast ratio for all text must be at least 4.5:1.
    • One exception - a lighter 3:1 contrast ratio will suffice if:
      • the text size is at least 14pt/19px and bold, or
      • the text is 18pt/24px and normal weight

    Avoid Images of Text

    Non-logo image assets do not contain pertinent graphical text. If an image contains pertinent or relavent text, alternative text should be provided.

    Avoid Using Color Alone

    Links must be visually identifiable via a non-color attribute e.g. an underline

    Integrating Accessibility Into Our Process

    General

    • Accessibility is a team effort!
    • Be proactive about accessibility
    • Retrofitting and remediation efforts are costly
    • Design, copy and UX must drive a11y requirements and direction from the beginning

    Delivery Management

    • Ensure dev oversight during wireframing and design process
    • Request dev sign off on client deliverables before delivery

    UX

    • Prescribe landmark and heading usage
    • Prescribe a "skip to content" link

    Design

    • Test and document color contrast in comps
    • Create a color usage page that outlines compliant color combination usage and color contrasts for each combo

    Copy

    • Capture accessibility text in copy decks
    • Work with clients to understand impact on CMS

    Tech

    • Follow standards and best practices
    • Test code in aXe or FireEyes
    • Manually test code frequently using screen readers
    • Mac users: set up VM's for JAWS, NVDA

    QA

    • Test code in aXe (free dev plugin for Chrome)
    • Manually test code using screen readers
