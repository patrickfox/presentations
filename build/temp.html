What is accessibility (a11y)?

For our clients:

  • Inclusivity
  • More sales, participation, improved brand perception
  • Avoiding lawsuits

For Razorfish:

  • Create efficient and innovative web experiences for all
  • Exceed the accessibility standards and best practices
  • Value-add, key differentiator
  • Being a good steward of the web
  • Looking out for our clients

For our end users:

  • Access to the conveniences of the web (e.g. banking, shopping, social)
  • Independence, self-reliance, participation

In a nutshell:

  • What can be seen or heard (color, imagery, icons, font weight/size, audio alerts) can be read/interpreted by a screen reader (and still make sense)
  • What can be heard can also be indicated to the user via visible text
  • What functions and interactions that can be performed with a mouse can be performed using a keyboard as well

Areas of difficulty

  • Inaccessible Flash content
  • CAPTCHA’s
  • Ambiguous links
  • Missing/improper alt text
  • Unexpected screen changes
  • Complex/difficult forms

  • Missing or improper headings
  • Too many links
  • Complex data tables
  • Lack of ‘Skip’ links
  • Inaccessible/missing search

Vision impairments

  • Partial to total blindness
  • Color blindness and other vision impairments

Hearing impairments

  • Partial to total deafness

Mobility impairments

  • Loss of mobility in arms/hands

Cognitive impairments

  • Attention deficit

Screen readers

Software that reads content, provides tools to navigate and understand UI

Top screen readers

  • JAWS
  • NVDA
  • VoiceOver (MacOS and iOS)
  • TalkBack (Android)

Basic Voiceover Commands

  • ⌘-F5 - toggle on/off
  • VO(ctrl + option) + left/right arrows - navigate through elements
  • VO + U - use arrows to view available elements
  • VO + A/B - start reading page from cursor or from top of page to cursor

Laws and Standards

Section 508 amendment to the Rehabilitation Act

Established a US government accessibility standard, requires that all US federal agencies ensure that their electronic and information technology are accessible per the Section 508 guidelines

WCAG2

W3C's international accessibility standard

  • levels of compliance: A, AA, AAA
  • WCAG2 Level AA covers the majority of Section 508

ARIA Specification

  • International, W3C spec
  • Addresses accessibility issues pertaining to dynamic and custom web UI
  • Defines a set of HTML attributes e.g. aria-label, role, aria-live

American's with Disabilities Act of 1990 (ADA)

A civil rights law that is intended to protect against discrimination based on disability

Specific to physical

National Federation of the Blind (NFB)

Blind and vision-impaired rights advocacy group, assists with high-profile lawsuits

Air Carrier Access Act (ACAA)

Mandates compliant, accessible web experiences for air carriers operating in the US by December 2015/2016

Accessibility for Ontarians with Disabilities Act, 2005 (AODA)

Canadian standard based on WCAG2 AA

Mandates timeline for government and public sector organization compliance

WCAG2 and Best Practice Overview

Structure/Layout

  • Ensure that content is accurately represented by HTML ("semantic HTML")
  • Headings and labels accurately describe topic or purpose (2.4.6)
  • Provide specific usage requirements for landmarks and headings
  • Provide a skip link to bypass repetitive header content (2.4.1)
  • Ensure meaningful focus order (2.4.3)
  • Ensure that pages can be found in multiple ways (e.g. via navigation, site map, site search) (2.4.5)

Heading and Landmark Usage

Landmarks and headings should be used sparingly to provide a high-level overview of page content

Proper use of landmarks and headings can greatly improve navigation and disoverability

Headings and Heading Order

  • Heading order should never decrease by more than 1:
  • Good: H2 > H3, H1 > H2, H4 > H2 (increases by 2, which is fine)
  • Not so good: H2 > H4, H1 > H4
  • There should be one H1 per page/view
  • Modal headings should start at H1

Landmark Usage

All page content should live within a landmark - avoid orphaned content

Landmark roles:

  • banner(required, 1 per page max) - header section container
  • main(required, 1 per page max) - main content section of page
  • complementary(optional) - secondary content related to main page content
  • contentinfo(required, 1 per page max) - footer section container
  • navigation - page/site navigation
  • search - enclosing search box container that contains search field; do not use directly on a form field

Behaviors

  • Provide timer, warnings and ability to extend time-based tasks or states/sessions (2.2.1)
  • Ensure that focus or input change does not change page context (3.2.1, 3.2.2)
  • Inform the user of any significant page updates (single-page apps)

Copy

  • Avoid use of sensory characteristic references (shape, size, visual location, orientation, sound, e.g. 'click button above') (1.3.3)
  • Provide alternate text when standard text is unavailable or insufficient to properly convey meaning
  • Avoid use of ambiguous interactive element copy (e.g. 'Click here', 'Read more') (2.4.4)

Forms

  • Provide suggestions to help users address form errors (3.3.3)
  • Notify users of form errors before submission, and allow them to review and correct any issues (3.3.4)
  • Form field errors, labels, requirements and instructions must be made apparent to the user (3.3.1, 3.3.2)

Video

  • Provide ability for users to control motion/audio (1.4.3, 2.2.2)
  • Video requires captions and audio description (1.2.1, 1.2.2, 1.2.3, 1.2.4, 1.2.5)

Visual Design

  • Avoid use of color alone to indicate meaning (1.4.1)
  • Ensure that minimum color contrast requirements are met (1.4.3)
  • Ensure visible focus state for interactive elements (2.4.7)
  • Ensure text can be sized up 2x without breaking UI (1.4.4)
  • Avoid use of graphical text in images (brand logos are main exception) (1.4.5)
  • Provide succinct, descriptive alt text for important images (Section 508)
  • Avoid flashing content (<3Hz and >55Hz) (2.3.1)

Tech

  • Page/phrase language is accurate and can be programmatically determined via proper use of the lang attribute (3.1.1, 3.1.2)
  • All functionality is operable via keyboard (2.1.1)
  • All img should have an alt attribute

Integrating Accessibility Into Our Process

General

  • Accessibility is a team effort!
  • Be proactive about accessibility
  • Retrofitting and remediation efforts are costly
  • Design, copy and UX must drive a11y requirements and direction from the beginning

Delivery Management

  • Ensure dev oversight during wireframing and design process
  • Request dev sign off on client deliverables before delivery

UX

  • Prescribe landmark and heading usage
  • Prescribe a "skip to content" link

Design

  • Test and document color contrast in comps
  • Create a color usage page that outlines compliant color combination usage and color contrasts for each combo

Development Tips

  • Minimize the use of ARIA tags
  • Don't overcomplicate your code

Provides essential accessibility tools:

  • Announce content using $.announce()
  • Focus on anything using $.access()

Use of Tabindex

What can receive focus?

  • Native interactive elements
  • Any element with tabindex="-1" or greater value
  • tabindex="-1" allows for programmatic focus only

Use of Tabindex

Best Practices

  • Do not use tabindex > 0
  • Rely on natural tab order
  • Use native interactive elements

Modals

Goal

  • Announce presence of modal and purpose
  • Keep user isolated in modal

Modals

Setup

  • H1 is first heading
  • role="dialog"
  • aria-labelledby to label modal

Modals

On Open

  • UI Isolation
  • place focus on first focusable
  • Esc key closes dialog (unless action required)

Modals

On Close

  • remove UI isolation
  • place focus back on modal trigger

Tooltips

  • Tooltips contain succinct, non-interactive, informational content only
  • On hover/focus, it's OK to display a tooltip if contents are read to user
  • E.g. button(role="tooltip" aria-label="Tooltip content")

Form Validation

Basics

  • Every form input requires a label (via label(for="id") or aria-label)
  • aria-required="true" to indicate required fields

Form Validation

Error/help messages

  • Visible error/help messages must be read upon interaction with field
  • Use aria-invalid="true" to indicate invalid fields
  • Use $.announce() to announce error messages to user

Collapsibles

Button

  • Toggle aria-expanded="true|false" to indicate expanded/collapsed state

Collapsibles

Content

  • display: none; to hide inactive content from screen readers

Popup Menu

Button

  • Use aria-haspopup="true" to indicate popup menu button
  • Use aria-expanded="true|false" to indicate current menu state

Popup Menu

Menu

  • Use display: none; to hide inactive menu content from screen readers
  • Emulate native menu keyboard navigation (arrow keys, esc, tab)

Buttons vs. Links

Use buttons and links for their intended purpose

  • Links: site navigation, external sites
  • Buttons: internal functionality

Don't use a(role="button") or button(role="link")

Don't create custom buttons from non-interactive elements
