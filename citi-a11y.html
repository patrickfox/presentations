Compliance Strategy and Goals

Opportunities and Inclusivity

  • Create efficient and innovative web experiences for all
  • Exceed the accessibility standards and best practices
  • Competetive value-add, key differentiator
  • Be a good steward of the web
  • Improve brand perception
  • Minimize legal risks

Building expertise

Continuous training and review
  • Assign an accessibility lead
  • Conduct periodic training
  • Continuous feedback via frequent code reviews
  • Become screen reader experts

Common accessibility issues

From WebAIM's Annual Screen Reader Survey (2014)

  • Inaccessible Flash content
  • CAPTCHA’s
  • Ambiguous links
  • Missing/improper alt text
  • Unexpected screen changes
  • Complex/difficult forms
  • Poor keyboard accessibility
  • Missing or improper headings
  • Too many links
  • Complex data tables
  • Lack of ‘Skip’ links
  • Inaccessible/missing search

Dev process and testing

General

  • Accessibility is a team effort
  • Be proactive about accessibility - don't put it off until QA
  • Build accessibility into each task
  • Review and test as you go
  • Retrofitting and remediation efforts are costly
  • Waiting until QA is the same as remediation

Design review and technical planning

  • Review and approve designs for accessibility early on
  • Create prototypes to address unknown situations

Automated testing

Make compliance testing part of the development process
  • Tests color contrast, code validation
  • aXe (Chrome plugin)
  • Total Validator (browser plug-in)
  • Tenon (a11y testing service)
  • Write custom scripts

Manual testing

Automated testing only accounts for ~30% of a11y issues
  • Ensure keyboard operablility
  • Ensure that UI and content structure are clear via screen reader

Screen reader/browser support

Goal: compliance across widest range of screen reader/browser combos, within reason...
  • Focus on most widely used screen reader/browser combos
    • Win: JAWS and IE, NVDA and FF
    • Android: TalkBack and FF
    • MacOS, iOS: VO and Safari
  • Not every feature will work the same across the combos
  • Document known issues for future reference

Screen reader/browser support

Report screen reader and browser defects to their manufacturers
  • [Apple Bug Reporter](https://bugreport.apple.com/)
  • [Google Chrome Bugs](http://dev.chromium.org/for-testers/bug-reporting-guidelines)
  • [NVDA Bugs](http://community.nvda-project.org/wiki/Issues)
  • [JAWS Bugs](http://www.freedomscientific.com/Forms/TechSupport)

Implementation Best Practices

Skip Navigation and Headings

  • Provide "Skip to content" link to bypass repetetive content
  • Use headings to create content hierarchy
    • Don't overuse headings
    • Use one H1 per view
    • Use headings to improve content navigation and understanding
    • Avoid links in headings
    • Visual size should not indicate heading use

Landmarks

Landmarks provide a high level map of the major content sections
  • All page content should live within a landmark
  • Every page should have one banner, main, contentinfo
  • Generic landmarks like navigation, complementary require a label

Landmarks

Landmarks provide a high level map of the major content sections
  • Search should use role="search"
  • HTML5 sectioning elements may be recognized as landmarks
    • aside is a complementary in VO by default
    • Is a landmark: use the accompanying role
    • Is not a landmark: use role="presentation"
  • Don't use form or application roles

Alternative Text

  • Improves understanding and accessibility for screen reader users
  • Use alt for images
  • aria-label for screen-reader only content
  • aria-labelledby, aria-describedby - use available content to label something

Focus Management

  • Care for user focus
  • Ensure that UI updates do not change user focus unless:
    • The focus change is intended and necessary
    • the focus change improves the user's understanding of a context change
    • E.g. modals, content is no longer available in new context

Tabindex

  • Do not hard code tabindex - rely on natural tab order
  • Need a clickable/user interactive element?
    • Use a button or a for intended purpose
    • Don't: div(role="button" tabindex="1")
    • Don't: a(role="button") or button(role="link")
  • Need to put focus somewhere in the UI that isn't focusable?
    • Use a script to manage focus and place focus temporarily
    • See $.access()

ARIA

Question the need for and minimize use of ARIA attributes
  • aria-required="false" - redundant
  • aria-invalid="false" - redundant
  • aria-hidden="false" - redundant, causes issues
Don't arbitrarily use ARIA attributes without testing

Announcing Content

Use a single live region to communicate pertinent messages
  • Multiple live regions complicates code and testing
  • Avoid live region roles: `role='status|log|alert|timer'`
  • Use to announce pertinent updates
    • Single-page app view changes
    • Any pertinent alerts/errors
Say anything w/ $.announce()

Form Validation

Basics

  • Every form input requires a label (via label(for="id") or aria-label)
  • aria-required="true" to indicate required fields
  • Avoid links in input labels

Form Validation

Error/help messages

  • Visible error/help messages must be read upon interaction with field
  • Use aria-invalid="true" to indicate invalid fields
  • Use $.announce() to announce error messages to user

Widget Keyboard Navigation

  • Emulate native browser functionality where possible
  • Use ARIA to provide state semantics

Modals

Goal

  • Announce presence of modal and purpose
  • Keep user isolated in modal

Modals

Setup

  • H1 is first heading
  • role="dialog"
  • aria-labelledby to label modal

Modals

On Open

  • UI Isolation - hide main content using aria-hidden="true"
  • Tab Fence - script keeps focus in modal
  • Place focus on first focusable element
  • Esc key closes dialog (unless action required)

Modals

On Close

  • Remove UI isolation
  • Place focus back on modal trigger

Tooltips

  • Tooltips contain succinct, non-interactive, informational content only
  • On hover/focus, it's OK to display a tooltip if contents are read to user
  • E.g. button(role="tooltip" aria-label="Tooltip content")
Tooltip example

Collapsibles

Button

  • Toggle aria-expanded="true|false" to indicate expanded/collapsed state

Collapsibles

Content

  • display: none; to hide inactive content from screen readers

Drop-Down Menus

Button

  • Use aria-haspopup="true" to indicate popup menu button
  • Use aria-expanded="true|false" to indicate current menu state

Drop-Down Menus

Menu

  • Use display: none; to hide inactive menu content from screen readers
  • Emulate native menu keyboard navigation (arrow keys, esc, tab)
